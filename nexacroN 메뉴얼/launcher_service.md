1.1 개발 배경

웹 페이지 내에서 외부 애플리케이션을 임베디드 형식으로 사용하는 방식은 액티브엑스 또는 플러그인 기술을 사용

합니다. 초기에는 HTML 코드만으로 처리하기 어려웠던 부분을 보완하면서 다양한 분야에서 활용되었습니다.

시간이 지나면서 플러그인 기술은 보안 허점을 드러냈고 공격 대상이 되면서 보안 이슈가 늘어나고 있습니다. 최근의

해킹 사례 중 많은 부분이 구형 웹브라우저에 연결된 액티브엑스를 통해 발생하고 있습니다. 이런 이유로 구글이나 마

이크로소프트에서는 새로 출시하거나 업데이트하는 제품에서는 액티브엑스 또는 플러그인 기능을 더는 지원하지 않

는 정책을 발표하고 있습니다.

NPAPI deprecation: developer guide

http://www.chromium.org/developers/npapi-deprecation

A break from the past, part 2: Saying goodbye to ActiveX, VBScript, attachEvent…

http://blogs.windows.com/msedgedev/2015/05/06/a-break-from-the-past-part-2-saying-goo

이런 이슈가 아니더라도 모바일을 비롯한 다양한 환경의 사용자가 늘어나면서 자바스크립트와 HTML 코드만으로 웹

애플리케이션을 만들고자 하는 요구가 늘어나고 있습니다. 특히 정부와 공공기관에서는 정책적으로 플러그인 기술을

배제하는 추세입니다.

하지만 운영하는 애플리케이션을 새로 개발하기에는 큰 비용과 기간이 필요하며 기업 운영에 부담을 줄 수 있습니다.

이런 문제를 해결하기 위해 투비소프트에서는 런처 서비스를 개발했습니다. 런처 서비스를 사용하면 액티브엑스 또

는 플러그인 기술을 사용하지 않아 웹브라우저의 보안 이슈를 감소할 수 있으며 기존에 개발된 애플리케이션을 그대

로 사용할 수 있습니다.

1.2 런처 서비스의 특징

● 액티브엑스나 플러그인 기술을 사용하지 않고 별도 서비스로 동작합니다.

● 운영체제 시작 시 자동으로 시작할 수 있는 윈도우 서비스 방식으로 런처를 운영합니다.

● 외부에서 접근할 수 없는 로컬 IP(127.0.0.1)을 활용해 필요한 통신을 처리합니다.

● 자바스크립트 표준 객체 표기법인 JSON 형식으로 데이터 송수신을 처리합니다.

제약 사항

6.1 운영체제 관련 제약사항

6.1.1 가상화 시스템 사용 시 문제가 발생할 수 있습니다

런처 서비스는 데스크탑 단위로 동작합니다. 로컬 데스크탑의 윈도우 서비스를 활용해 웹페이지에서 전달받은 통신

데이터를 분석한 후 적절한 동작을 취하는 방식으로 동작합니다. 물리적으로 같은 데스크탑이라 하더라도 일부 가상

화 시스템에서는 알려지지 않은 문제가 발생할 수 있습니다.

6.1.2 메시지 박스로 질의하는 동작은 지원하지 않습니다.

런처 서비스는 데스크탑의 백그라운드 서비스로 동작하기 때문에 메시지 박스를 띄워 질의하는 형태는 지원하지 않

습니다.

6.1.3 다중 로그인 상태에서 정보가 전달되지 않을 수 있습니다.

여러 사용자가 로그인한 상태에서는 런처 서비스에서 어떤 사용자가 로그인했는지 알 수 없기 때문에 정보 전달을 제

대로 하지 못하고 애플리케이션을 구동하기 됩니다.

예를 들어 윈도우 운영체제에서 A 사용자가 로그인한 상태에서 로그아웃을 하지 않고 B 사용자로 계정을 전환하는

경우 프로세스에서는 두 계정의 프로그램이 모두 살아있는 상태가 되어 런처 서비스에서 어느 계정에 명령을 전달하

고 있는지 파악할 수 없습니다.

이를 방지하기 위해서는 기존 사용자 계정을 로그아웃 후 다른 사용자가 로그인 하는 형식으로 사용해야 합니다.

6.2 웹브라우저 관련 제약사항

6.2.1 특정 브라우저 사용 시 사용하는 웹페이지를 신뢰 사이트로

등록해야 합니다.

인터넷 익스플로러(IE) 9 이하 버전에서는 서로 다른 주소 접근(Cross Domain)에서 자바스크립트 XMLHttpReque

st.open 메서드 사용 시 '액세스를 허용하지 않는다'는 에러 메시지가 나올 수 있습니다. 이런 경우 통신 데이터 자체

를 전송하지 않습니다. 이런 경우 윈도우 서비스와 정상적으로 통신하기 위해서는 웹페이지 주소를 신뢰할 수 있는 사

이트로 등록해야 합니다.

6.2.2 지정된 JSON 형식과 다를 경우 문제가 발생할 수 있습니다.

JSON 형식으로 정보를 주고 받기 때문에 대소문자를 구분해서 설정하지 않으면 문제가 발생할 수 있습니다.

6.2.3 구글 크롬에서 HTTPS 통신 사용 시 특정 옵션을 활성화해주

어야 합니다.

런처 서비스 1.1.0.24 이상 버전에서는 해당 제약사항이 해소됐습니다.

윈도우 7 운영체제 사용 시에는 추가 설정이 필요합니다.

구글 크롬 88 미만 버전과 런처 서비스 1.1.0.24 미만 버전을 사용해야 하는 경우 해당 옵션을 사용할

수 있습니다. 구글 크롬 88 이상 버전에서는 해당 옵션을 지원하지 않기 때문에 런처 서비스를 1.1.0.24
